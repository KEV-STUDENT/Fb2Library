# Architecture Decision Record

# ADR-001: Выбор Clean Architecture

## Контекст
Приложение читает FB2/ZIP и хранит в БД. Логика парсинга может меняться.

## Решение
Разделить на Domain (сущности), Application (сценарии), Infrastructure (файлы/БД).

## Последствия
✅ Независимость от UI  
✅ Легко тестировать  
❌ Больше проектов  



# ADR-002: Выбор TDD

## Контекст
Освоить написание юнит-тестов

## Решение
Использовать стек [xUnit + FluentAssertions + NSubstitute](./TESTS.md)

## Последствия
✅ Новый функционал не ломает старый - TDD создает страховочную сеть. Каждый баг сначала превращается в красный тест, а потом чинится.  
✅ Вы четко описываете правила игры до кодирования.  
✅ Код становится «тестируемым» только если он модульный и слабосвязанный. TDD заставляет вас сразу делать инъекцию зависимостей и интерфейсы, а не лепить «божественные объекты».  
✅ TDD дисциплинирует.  
✅ Если вы боитесь трогать старый легаси-код, потому что «все развалится», TDD дает уверенность: зеленые тесты — разрешение на правку.  

❌ UI-разработка (верстка, дизайн) - Здесь работают скриншотные тесты или ручной осмотр, но не классический TDD.  
❌ Тривиальный код без логики - Геттеры, сеттеры, простые прокси-методы, CRUD-контроллеры без валидации. Если вероятность ошибки равна нулю, тест не нужен.  
❌ Тестируется реализацию, а не поведение (при каждом рефакторинге падает 50 тестов).  
❌ Выполнение тестов занимает больше нескольких секунд на юнит (перестают их запускать).  
❌ Сжатые дедлайны и отсутствие культуры - Если команда никогда не писала тесты, внедрение TDD «с завтрашнего дня» убьет скорость в 2-3 раза без гарантии качества.  



# ADR-003: Еип Id для сущностей

## Контекст
Выбрать тип Id и место создания для новых экземпляров (в БД или нет)

## Решение
Использовать GUID как Id, генерация вне БД

## Последствия
✅ Агрегат полноценен с момента создания — можно сразу публиковать Domain Events с Id  
✅ Нет state "недосозданного" агрегата — не нужен nullable Id  
✅ Проще тестировать — не нужна БД для получения Id  
✅ Можно использовать в других агрегатах до сохранения:  
csharp  
var book = Book.Create("Title");  
var review = Review.Create(book.Id, "Great!"); // Id уже есть!  
✅ Подходит для Event Sourcing — события пишутся до сохранения состояния  

❌ Сложнее с автоинкрементными Id (если нужны)  
❌ GUID занимает больше места в индексах, чем последовательный int/long

# Architecture Decision Record

# ADR-001: Выбор Clean Architecture

## Контекст
Приложение читает FB2/ZIP и хранит в БД. Логика парсинга может меняться.

## Решение
Разделить на Domain (сущности), Application (сценарии), Infrastructure (файлы/БД).

Сервисы-оркестраторы являются частью слоя Application. Они не являются бизнес-логикой, а реализуют координацию сценариев использования (use case coordination).  
Их интерфейсы определяются в слое Application, а используются они в слое Infrastructure.

## Последствия
✅ Независимость от UI  
✅ Легко тестировать  
❌ Больше проектов  



# ADR-002: Выбор TDD

## Контекст
Освоить написание юнит-тестов

## Решение
Использовать стек [xUnit + FluentAssertions + NSubstitute](./TESTS.md)

## Последствия
✅ Новый функционал не ломает старый - TDD создает страховочную сеть. Каждый баг сначала превращается в красный тест, а потом чинится.  
✅ Вы четко описываете правила игры до кодирования.  
✅ Код становится «тестируемым» только если он модульный и слабосвязанный. TDD заставляет вас сразу делать инъекцию зависимостей и интерфейсы, а не лепить «божественные объекты».  
✅ TDD дисциплинирует.  
✅ Если вы боитесь трогать старый легаси-код, потому что «все развалится», TDD дает уверенность: зеленые тесты — разрешение на правку.  

❌ UI-разработка (верстка, дизайн) - Здесь работают скриншотные тесты или ручной осмотр, но не классический TDD.  
❌ Тривиальный код без логики - Геттеры, сеттеры, простые прокси-методы, CRUD-контроллеры без валидации. Если вероятность ошибки равна нулю, тест не нужен.  
❌ Тестируется реализацию, а не поведение (при каждом рефакторинге падает 50 тестов).  
❌ Выполнение тестов занимает больше нескольких секунд на юнит (перестают их запускать).  
❌ Сжатые дедлайны и отсутствие культуры - Если команда никогда не писала тесты, внедрение TDD «с завтрашнего дня» убьет скорость в 2-3 раза без гарантии качества.  



# ADR-003: Tип Id для сущностей

## Контекст
Выбрать тип Id и место создания для новых экземпляров (в БД или нет)

## Решение
Использовать GUID как Id, генерация вне БД

## Последствия
✅ Агрегат полноценен с момента создания — можно сразу публиковать Domain Events с Id  
✅ Нет state "недосозданного" агрегата — не нужен nullable Id  
✅ Проще тестировать — не нужна БД для получения Id  
✅ Можно использовать в других агрегатах до сохранения:  
csharp  
var book = Book.Create("Title");  
var review = Review.Create(book.Id, "Great!"); // Id уже есть!  
✅ Подходит для Event Sourcing — события пишутся до сохранения состояния  

❌ Сложнее с автоинкрементными Id (если нужны)  
❌ GUID занимает больше места в индексах, чем последовательный int/long  



# ADR-004: Взаимодействие между компонентами

## Контекст
В рамках развития системы необходимо стандартизировать взаимодействие между компонентами приложения. Существующий подход с прямой инъекцией сервисов приводит к:  
* Сильной связанности (high coupling) между слоями, особенно между контроллерами/API(Infrastructure) и слоем бизнес-логики(Application).  
* Разрастанию конструкторов («толстые» контроллеры) из-за большого количества внедряемых зависимостей.  
* Сложности добавления сквозной функциональности (логирование, валидация, управление транзакциями), что часто приводит к дублированию кода.  
* Затрудненному модульному тестированию компонентов UI/API, так как требуется мокать множество интерфейсов, не относящихся к тестируемому сценарию.

## Решение
Использовать библиотеку MediatR как основного Посредника для реализации паттернов CQRS (Command Query Responsibility Segregation) и шины событий (Event Bus).

## Детали реализации
Запросы (Queries) и Команды (Commands): Все взаимодействия из слоя представления (контроллеры, обработчики gRPC, консольные команды) к слою приложения инкапсулируются в объекты IRequest<TResponse> (запросы/команды). Контроллеры зависят только от интерфейса ISender.  
Обработчики (Handlers): Каждый запрос или команда имеет ровно один обработчик (IRequestHandler<TRequest, TResponse>), содержащий всю необходимую бизнес-логику.  
События домена/интеграции: Для реализации побочных действий, не влияющих на основной поток, используются INotification и INotificationHandler. Например, после создания заказа (OrderCreatedEvent) публикуется уведомление, на которое могут подписаться обработчики для отправки email и обновления склада, не задерживая основной HTTP-ответ.  
Pipeline Behaviors: Сквозная функциональность (валидация входных команд с помощью FluentValidation, логирование, замер времени выполнения) реализуется как IPipelineBehavior, оборачивающий вызовы обработчиков.

## Альтернативы
Прямые вызовы сервисов: Было решено отказаться, так как это приводит к проблемам, описанным в разделе «Контекст».  
Собственная реализация Посредника: Отвергнуто, так как MediatR является зрелым, легковесным и широко поддерживаемым решением без зависимостей, что избавляет от необходимости писать инфраструктурный код.  
Коммерческие шины (NServiceBus, MassTransit): Избыточны для внутрипроцессного взаимодействия. Однако MediatR не исключает их совместного использования: INotificationHandler может публиковать сообщения во внешнюю шину для межсервисного взаимодействия.

## Последствия
✅ Слабая связанность: Слой представления полностью отделен от слоя бизнес-логики. Контроллеры становятся «тонкими» и зависят только от одного интерфейса (ISender или IMediator).  
✅ Соблюдение Single Responsibility Principle (SRP): Каждый обработчик отвечает строго за один сценарий использования. Код становится узкоспециализированным.  
✅ Тестируемость: Модульное тестирование упрощается. Для тестирования контроллера достаточно мокать ISender. Для тестирования бизнес-логики — тестируется изолированный Handler с моками только тех зависимостей, которые нужны ему.  
✅ Расширяемость через Pipeline: Добавление новой сквозной логики (кеширование, повторные попытки при сбоях, авторизация) не требует модификации существующих обработчиков.

❌ Повышенная сложность навигации: Переход от контроллера к коду, который реально выполняется, требует дополнительного шага (поиск обработчика по имени команды/запроса, что непривычно для новичков). Требует дисциплины в именовании.  
❌ Неявная логика: Злоупотребление INotification может привести к «магическому» поведению, когда причины побочных действий трудно отследить во время отладки.  
❌ Риск оверинжиниринга: Для простых CRUD-операций без дополнительной логики добавление отдельных Command/Query/Handler классов создает избыточный объем кода.  
❌ Производительность: Незначительные накладные расходы на сериализацию/десериализацию внутри памяти, разрешение типов и выполнение цепочки Pipeline Behaviors. В абсолютном большинстве случаев являются пренебрежимо малыми.



# ADR-005: Разделение бизнес-логики между Сервисами и MediatR

## Контекст
После принятия ADR-004 (использование MediatR для взаимодействия между компонентами) возник вопрос о роли классических Сервисов в архитектуре приложения.  

Полный отказ от Сервисов в пользу «чистого» MediatR создаёт следующие неудобства:  
* Контроллеры вынуждены напрямую зависеть от IMediator, что смешивает инфраструктурные заботы (отправка сообщений) с точкой входа API.  
* Сложная оркестрация (вызов нескольких команд, управление транзакциями, координация саг) ложится либо на контроллер, либо размазывается между обработчиками через вложенные уведомления, что ухудшает читаемость.  
* С другой стороны, возврат к «толстым» Сервисам, содержащим всю бизнес-логику, сводит на нет преимущества CQRS и pipeline behaviors, полученные в ADR-004.

## Решение
Использовать гибридный подход с двумя допустимыми моделями взаимодействия:  
 Сервисы как Оркестраторы + MediatR как исполнители бизнес-логики.  
 Для простых задач использовать MediatR без сервиса.  
```
┌──────────────────────────────────────────────────────┐
│ Infrastructure (Контроллеры)                         │
│                                                      │
│  ┌─────────────┐          ┌──────────────────────┐   │
│  │ Модель A:   │          │ Модель B:            │   │
│  │ Controller  │          │ Controller           │   │
│  │    │        │          │    │                 │   │
│  │    ▼        │          │    ▼                 │   │
│  │ Service     │          │ MediatR напрямую     │   │
│  │    │        │          │                      │   │
│  │    ▼        │          │                      │   │
│  │ MediatR     │          │                      │   │
│  └─────────────┘          └──────────────────────┘   │
├──────────────────────────────────────────────────────┤
│ Application                                          │
│  ├── Services (оркестраторы, только для модели A)    │
│  ├── Commands / Queries                              │
│  ├── Handlers (бизнес-логика)                        │
│  └── Notifications / Handlers (побочные действия)    │
└──────────────────────────────────────────────────────┘

```

## Альтернативы
Чистый MediatR (без Сервисов): Контроллеры напрямую зависят от IMediator. Отвергнуто, так как размывает границу между слоями Infrastructure и Application, превращая контроллер в оркестратор.  
Чистые Сервисы (без MediatR): Вся логика в методах Сервиса. Отвергнуто в ADR-004, так как лишает преимуществ CQRS и Pipeline Behaviors.  
Сервисы, содержащие бизнес-логику + MediatR как прокси: Отвергнуто как нарушающее Single Responsibility Principle. Приводит к дублированию логики и путанице в том, где искать код.  

## Последствия
✅ Чёткое разделение ответственности: Сервис = «что и в каком порядке сделать», Handler = «как именно сделать».  
✅ Чистые контроллеры: Зависят только от интерфейса Сервиса. Не знают о существовании MediatR.  
✅ Тестируемость: Оркестрация тестируется моком IMediator. Бизнес-логика тестируется изолированно в Handler. Побочные действия — в NotificationHandler.  
✅ Готовность к росту: Сервис-оркестратор легко превращается в координатор распределённой саги без изменения контроллеров или обработчиков.  
✅ Прагматичность: Простые операции остаются простыми, сложные получают необходимую структуру.  
✅ Единый механизм Pipeline Behaviors: Валидация и логирование работают одинаково в обеих моделях.  
✅ Отсутствие «мёртвого кода»: Не создаются сервисы-прокладки ради соблюдения формального правила

❌ Дисциплина команды: Требуется строгое соблюдение правила «Сервис не содержит бизнес-логики», иначе подход вырождается в бессистемное смешение стилей.  
❌ Два места для внедрения зависимостей: Разработчик должен решить, где разместить новую функциональность — в Сервисе или в Handler.  
❌ Неоднородность в контроллерах: Один контроллер может использовать и IMediator, и IService. Требуется самодисциплина.  
❌ Рефакторинг при росте: Если простая CRUD-операция со временем обрастает логикой, потребуется выделить Сервис и переключить контроллер на него.

Связанные решения
ADR-004: Взаимодействие между компонентами (использование MediatR)



# ADR-006: Управление зависимостями через Dependency Injection

## Контекст
Приложение состоит из множества компонентов, распределённых по слоям согласно ADR-005 и ADR-004. Компоненты имеют зависимости друг от друга (репозитории, сервисы-оркестраторы, MediatR, инфраструктурные сервисы).

Необходим единый способ:
* разрешения зависимостей между компонентами,
* управления временем жизни объектов,
* подмены реализаций для тестирования,
* соблюдения принципа инверсии зависимостей (DIP) из SOLID: «Модули верхнего уровня не должны зависеть от модулей нижнего уровня. Оба должны зависеть от абстракций».

Ручное управление зависимостями (new-инг объектов вручную) приведёт к:
* жёсткой связанности между слоями,
* невозможности модульного тестирования с моками,
* дублированию кода создания объектов,
* нарушению Layer Dependency Rules (внутренние слои будут знать о внешних).

## Решение
Использовать встроенный DI-контейнер Microsoft.Extensions.DependencyInjection как единственный механизм управления зависимостями во всём приложении.

Ключевые принципы:
1. Интерфейсы объявляются во внутреннем слое, реализации — во внешнем.
2. Запрещено использовать Service Locator паттерн.
3. Внедрение через конструктор (Constructor Injection) как единственный способ.
    * Никакого Property Injection.
    * Никаких статических ссылок на контейнер.
4. Соблюдение явных зависимостей (Explicit Dependencies Principle).
5. Класс запрашивает в конструкторе только то, что ему действительно нужно.

## Последствия
✅ Тестируемость: Любой компонент легко тестируется изолированно с подменой зависимостей на моки/стабы.  
✅ Слабая связанность: Классы зависят от абстракций, а не от конкретных реализаций. Замена реализации репозитория (EF Core → Dapper → InMemory для тестов) не требует изменений в бизнес-логике.  
✅ Централизованное управление временем жизни: DI-контейнер гарантирует корректный Scoped в рамках HTTP-запроса, предотвращая утечки и сайд-эффекты между запросами.  
✅ Соблюдение архитектурных границ: Контейнер конфигурируется на верхнем уровне, соблюдая правило «зависимости направлены внутрь».  
✅ Автоматическая регистрация через сборки: MediatR автоматически находит и регистрирует все Handlers и Behaviors, исключая ручной бойлерплейт.

❌ Неявное поведение при ошибках: Если забыли зарегистрировать зависимость, ошибка возникает в рантайме, а не на этапе компиляции. Лечится интеграционными тестами.  
❌ Магические строки и конфигурации: Некоторые DI-контейнеры поощряют конфигурацию через XML/JSON. В нашем случае — только код, что минимизирует проблему.  
❌ Усложнение отладки: Стек вызовов становится глубже из-за прокси и декораторов контейнера. Не критично при стандартном подходе.  
❌ Зависимость от фреймворка: При переносе на другой хост (например, с ASP.NET на консольное приложение) потребуется адаптировать регистрацию DI.  

Связанные решения
* ADR-004: Взаимодействие между компонентами через MediatR
* ADR-005: Разделение бизнес-логики между Сервисами и MediatR
